faq:
  - q: "What's the most important factor in choosing an image compressor?"
    a: "It depends on use case. For one-off compressions: ease of use and privacy (browser-side). For batch processing: throughput and metadata preservation. For production assets: maximum compression quality (TinyPNG or Squoosh's MozJPEG). For privacy-sensitive content: browser-side only (Pickrack, Squoosh)."
  - q: "How much can a 4MB photo realistically compress?"
    a: "For a typical iPhone or DSLR photo at JPG quality 80%, expect 70-85% reduction (4MB → 600KB-1.2MB) without visible quality loss. For WebP or AVIF at the same quality, another 25-50% reduction (down to 200-700KB). Beyond that, quality starts being visible."
  - q: "Is server-side compression always better?"
    a: "No. Server-side compressors like TinyPNG use proprietary algorithms (custom MozJPEG, fine-tuned PNG) that beat browser-side by 10-20%. But the gap is small. For privacy-sensitive content, browser-side is preferred — the 10-20% extra compression isn't worth the upload."
  - q: "Why does TinyPNG charge?"
    a: "TinyPNG was the first AI-driven web image compressor (2014). They invested in proprietary algorithms and brand recognition. The free tier is generous for casual use (500KB/file, 20 files/month), and the paid tier ($39/year) is competitively priced for professionals. They have legitimate engineering value — but free alternatives have caught up enough that paying isn't required for most users."
  - q: "Will any compressor preserve EXIF metadata?"
    a: "Most strip it as a side effect of canvas re-encoding. ImageOptim has a 'preserve metadata' toggle. JPEGmini preserves EXIF by default. For most web use, stripping EXIF is actually preferred (privacy + smaller files). For archival purposes, use ImageOptim or run exiftool to copy metadata after compression."
  - q: "Is browser-side compression slower?"
    a: "Modern browsers (post-2022) are very fast at canvas operations. A 4MB photo compresses in 1-3 seconds in browser, similar to server-side. The bottleneck for cloud compressors is upload time — for a 4MB file on average broadband, upload + processing + download = ~5-10 seconds. Browser-side stays at ~1-3 seconds because there's no upload."
  - q: "What about animated GIF compression?"
    a: "Most JPG/PNG compressors don't handle animated GIFs well. Use ezgif.com (cloud) or convert GIF to animated WebP (typically 30-50% smaller, supported by all modern browsers). Pickrack doesn't handle GIF compression yet — on the roadmap."
  - q: "Can I bulk compress 1000+ photos?"
    a: "Yes. ImageOptim (Mac desktop) handles thousands. Pickrack handles batches up to ~50 in one go (browser memory limit). For 1000+ files at once, use a desktop tool or scripted CLI (Sharp via Node.js, Squoosh CLI). Browser-based tools are best for batches under 100."

## Test methodology

Same 4MB JPG photo (4032×3024, iPhone shot of a colorful market scene). Same target: visually lossless reduction. Compared:

- File size after compression
- Visible quality at 100% zoom
- Batch capability
- Upload requirement (privacy)

## 1. Pickrack Image Compressor 🥇

[**pickrack.com/tools/image/image-compressor**](https://pickrack.com/tools/image/image-compressor/)

- **Result**: 4MB → 1.2MB at JPG 80%, or 4MB → 850KB at WebP 80%
- **Quality**: 4/5 — invisible loss at 100% zoom
- **Privacy**: Browser-side (Canvas API + toBlob)
- **Batch**: Yes, up to ~50 at once, ZIP download
- **Cost**: Free, unlimited

Pros:
- Real-time before/after with quality slider
- Format conversion built in (compress AND convert in one step)
- No signup, no daily limit
- Works on any device with a browser

Cons:
- Browser canvas encoder is ~10-15% less efficient than TinyPNG's proprietary algorithm
- No AVIF output yet (input only)
- Strips EXIF (privacy benefit, but inconvenient if you need metadata)

**Verdict**: Best free choice for everyday use. Can handle most photographers' batch needs.

## 2. Squoosh (Google Chrome team) 🥈

[**squoosh.app**](https://squoosh.app/)

- **Result**: 4MB → 1.05MB at MozJPEG 80% (best browser-side compression)
- **Quality**: 5/5 — uses MozJPEG, the gold-standard JPEG encoder
- **Privacy**: Browser-side
- **Batch**: One image at a time
- **Cost**: Free, unlimited

Pros:
- MozJPEG, MozJPEG, AVIF, WebP, OxiPNG — multiple compression engines
- Fine-grained quality controls (chroma sampling, progressive, etc.)
- Open source (Apache 2.0)
- Side-by-side preview at any zoom

Cons:
- One image at a time (no batch UI)
- Engineering-built UX (less polished for casual users)
- AVIF encoding is slow (10-30 seconds per photo)

**Verdict**: Best for power users who want maximum manual control on a single hero image.

## Avoid this trap

**Don't compress an already-compressed image.** Each compression cycle adds artifacts. If you compressed once at JPG 80% and the file is "still too big," the right move is to:

1. Find the ORIGINAL (RAW, PSD, source PNG)
2. Compress from original to a smaller target

Re-compressing the JPG 80% output to JPG 70% gives smaller files but visibly worse quality.

## Compression checklist

Before publishing web images:

- [ ] Resize to max dimensions needed (1920px hero, 800px body, 400px thumbnail)
- [ ] Choose format (WebP for new sites, JPG for legacy)
- [ ] Compress at 80-85% quality
- [ ] Verify visible quality at 100% zoom
- [ ] Test on real device (laptop full-screen + mobile)
- [ ] Strip EXIF (privacy + smaller)
- [ ] Use `<picture>` with format fallbacks for cross-browser
